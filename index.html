01 · 协议介绍
What is A2A?

什么是 A2A Protocol

Agent2Agent（A2A）Protocol 是一个开放标准，旨在实现不同 AI Agent 之间的无缝通信与协作。 它为构建在不同框架上、由不同供应商开发的 Agent 提供了通用语言。

🤖

核心问题

AI Agent 通常由不同开发者、不同框架、部署在不同组织，彼此难以直接协作，形成信息孤岛。

🔓

A2A 的解法

提供标准化通信协议，让来自不同来源的 Agent 联合起来，共同完成复杂任务，打破信息孤岛。

📋

技术基础

复用 HTTP、JSON-RPC 2.0、Server-Sent Events 等成熟标准，降低学习成本和实现难度。

🏛️

治理

最初由 Google 开发，现已捐赠给 Linux 基金会进行维护和管理，保持开放中立。

01 · 设计原则
Design Principles

五大核心设计原则

⚡

简单性 Simplicity

复用 HTTP、JSON-RPC 2.0、Server-Sent Events 等经过广泛验证的标准，降低学习成本和实现难度。

🏢

企业级就绪 Enterprise-Ready

充分考虑身份验证、授权、安全性、隐私保护、追踪和监控等企业级需求，与成熟实践保持一致。

⏳

异步优先 Async-First

专为长时间运行的任务和 Human-in-the-Loop 交互而设计，能够优雅处理长时间运行的操作。

🎨

模态无关 Modality-Agnostic

支持多种内容类型：文本、音频/视频（文件引用）、结构化数据/表单，乃至嵌入式 UI 组件。

🔒

不透明执行 Opaque Execution

Agent 基于声明的能力和交换的信息协作，无需共享内部思考过程、计划或工具实现细节。

02 · 基本概念
Core Participants

三种核心参与者

👤

User（用户）

最终终端用户，通过与 A2A Client 交互发起请求。不直接参与 A2A 协议通信。

📡

A2A Client（客户端）

代表用户或其他系统发起请求的应用程序或 Agent。充当协调器，调度和整合多个远程 Agent 的能力。

🖥️

A2A Server（远程 Agent）

暴露 A2A 兼容端点的 Agent 或 Agentic 系统。发布 Agent Card，处理任务，提供响应。

User → 发起请求 → A2A Client → A2A 协议 → A2A Server （可继续调用其他 Server）
02 · 通信元素
Communication Elements

五大通信元素

🪪 Agent Card

Agent 的数字名片，JSON 格式元数据。包含身份、端点、能力列表、认证要求。发布于 /.well-known/agent-card

💬 Message

一次通信回合的基本单位。含唯一 messageId、角色标识（user/agent），承载一或多个 Part。

📋 Task

核心工作单元，具有唯一 ID 和清晰生命周期。用于长时间运行操作，通过 contextId 分组形成会话。

🧩 Part

最基本的内容容器，使用 oneof 结构。类型：TextPart / FilePart / DataPart

📦 Artifact

任务处理中生成的有形交付物（文档、图像、代码等）。含唯一 artifactId，支持增量流式传输。

02 · 交互机制
Interaction Mechanisms

三种交互模式

⚡ 同步消息

适用于即时、自我包含的交互，Agent 可立即返回结果，无需维护复杂状态。

→sendMessageClient 发送请求
←Message 对象直接返回

🌊 流式传输

通过 sendMessageStream 实现，允许实时接收任务更新。

→sendMessageStream
←Task 对象（开始）
←StatusUpdateEvent×N
←ArtifactUpdateEvent×N
←final=true（结束）

📨 推送通知

长时间运行或断开连接场景，通过 webhook 实现服务器主动推送通知。

→配置 webhook URL
←Task 状态变化时自动推送
←StreamResponse 格式

03 · 通信流程
Communication Flow

五步通信流程

1

Agent 发现

客户端通过 HTTP GET 请求获取目标 Agent 的 Agent Card，通常发布在 /.well-known/agent-card 端点。

2

身份验证

解析 securitySchemes 字段，根据 authorizationUrl 和 tokenUrl 向授权服务器请求 JWT 令牌。

3

发送消息

根据 Agent Card 中的 API 端点，发送 POST 请求（携带 JWT 令牌），请求体包含 Message 对象及任务描述。

4

处理与响应

Agent 可能创建 Task（异步模式）或直接返回 Message（同步模式）。异步任务在后台继续处理。

5

接收更新

三种方式获取异步任务更新：轮询（Get Task）· 流式订阅（持久连接）· 推送通知（webhook）

04 · 生命周期
Lifecycle & Context

多轮交互与上下文管理

🔑 contextId 的作用

逻辑分组多个 Task 和 Message，提供连续性。服务器负责生成；客户端提供则尽量复用。用于管理内部 LLM 上下文，确保多轮对话连贯性。

🔄 任务不可变性原则

任务达到终态后不可重启。后续细化须在同一 contextId 内发起新任务，可通过 referenceTaskIds 引用原任务。

💬 INPUT_REQUIRED 交互

Agent 需要额外信息时将状态设为 INPUT_REQUIRED，客户端检测后发送包含所需信息的跟进消息，任务继续处理。

⚡ 并行跟进

同一 contextId 内支持并行任务。客户端可在前置任务完成后立即创建依赖任务，实现高效并行工作流。

📦
产物版本管理（客户端驱动）
Agent 不维护产物变更历史，客户端负责追踪。服务 Agent 在生成细化版本时应使用一致的 artifact-name，方便客户端追踪最新版本。

沙盒经济两大维度

AI 代理沙盒经济由两个关键维度描述，这两个维度决定了经济风险的性质与可控程度

🌱 涌现型 (Emergent)

技术广泛采用的事实结果，自发形成。当前轨迹正指向这一方向——无规划的大型可渗透沙盒经济的意外涌现

🏗️ 刻意设计型 (Intentional)

为安全实验等目的蓄意构建。需要集体行动协调，因为渗透性是集体属性，非任何单一行为者可控

🔓 可渗透 (Permeable)

与既有人类经济可自由交互和交易。沙盒危机可能蔓延至真实经济，类似 2010 年"闪崩"

🔒 不可渗透 (Impermeable)

与人类经济密封隔离。防止不稳定性溢出，但需适当护栏将可渗透沙盒转为相对不可渗透

⚡ 核心挑战不是"是否"创建此生态系统，而是如何架构它使其可引导、安全、与用户及社区目标对齐

虚拟代理经济 · 分配
Distribution & Fairness

公平资源分配

🏛️ 拍卖机制

基于 Dworkin 分配正义理论：拍卖代理可利用的共享资源（算力、数据集、优先执行槽），而非代理本身。每位用户获得等量初始虚拟货币，确保平等购买力和协商能力。

✅ 嫉妒测试 (Envy Test)

公平标准：每个人的代理获得为其偏好量身定制的资源包，使得没有用户会更偏好他人的资源包+剩余货币。结果同时满足偏好敏感性和禀赋不敏感性。

🌐 社会选择理论

福利函数可建立社会结果的偏序关系，不仅考虑"好处"分配，也考虑"坏处"（外部性、风险）——包括 AI 代理的碳足迹。

能力不平等问题

研究表明更有能力的 AI 助手在协商中更成功。拥有最强代理的用户将获得不成比例的优势——形成危险的正反馈循环：经济优势 → 购买更强算力 → 进一步优势。

解决路径
→等额初始虚拟货币禀赋
→去中心化分布式分配机制
→多元公平概念的本地化适配
→监管框架确保规则合规

Privacy & Oversight

零知识证明与混合监督

🔐 零知识证明 (ZKPs)

允许代理证明某陈述为真，而无需披露使其为真的任何底层信息——实现隐私保护的同时确保可验证性

选择性披露
在谈判中证明有足够资金完成购买，而不暴露总预算——防止掠夺性定价
匿名凭证
证明属于某社区（如"X 社区居民"）而不暴露具体身份，防止跨场景追踪
不可链接性
每次交互生成全新证明，使观察者难以关联代理随时间的活动，打破信息茧房形成链条

🛡️ 混合多层监督

传统人工监督模型不足以应对机器速度的代理经济。需要一个结合 AI 实时监控与人类专家判断的混合分层系统

1
自动化 AI 监督员
实时监控市场活动，执行基本规则，标记欺诈/操纵/系统风险异常
2
自动裁决系统
对账户/交易临时冻结，收集相关数据供审查
3
人类专家审查
仅最复杂、新颖、高风险案件升级至此，确保人类注意力聚焦于最需之处

由公民、NGO、企业及公共行政机构发行，通过传统卡片系统、移动支付、区块链等技术实现。已有时间银行、普遍基本收入实验等实例。

🔗 与算力资源挂钩

社区货币可与共享算力资源挂钩——考虑到推理时扩展定律，复杂任务需要更多算力和更高环境成本。地理负载均衡机制可帮助公平分配环境影响。

虚拟代理经济 · 风险
Risks & Limitations

主要风险与局限

🪤

代理陷阱 (Agent Traps)

网站、数字元素或恶意输入蓄意颠覆代理操作完整性。通过越狱或对抗性提示利用代理指令跟随能力，诱导未授权支出或合同协议。

🔓

隐私与操纵风险

代理谈判时可能暴露用户偏好、策略或资源等敏感信息。恶意行为者可通过扩大信息不对称来剥削用户。

💼

劳动力替代

先进 AI 代理能自动化推理、规划和通信等大量非常规认知任务，从法律、会计到软件开发，加速工作极化与收入不平等。

📊

能力不平等加剧

拥有资本的人能购买更强大代理，在谈判中获得更大优势——形成算法强化的阶级结构，威胁市场公平性和流动性。

🌊

经济不稳定性

代理市场中的相互连接和快速反馈循环可能引发不可预见的灾难性涌现行为，如大规模虚拟代理经济崩溃蔓延至真实经济。

⚖️

责任归属难题

传统法律框架难以在代理创建者、部署者和用户之间分配责任。协调的多代理系统需要类似公司责任的群体代理法律模型。

虚拟代理经济 · 建议
Recommendations

五大关键建议

1
建立明确的责任与问责法律框架
新法律模型应将协调的多代理系统视为单一责任实体（类似公司责任），而非尝试追溯至单个代理——提供更稳健的集体行动问责路径
2
制定开放的代理互操作性和通信标准
碎片化数字生态系统将根本性限制代理经济潜力。A2A、MCP 等开放通用标准是防止"围墙花园"的基础——代理须能无缝发现能力、协商条款、安全交易
3
构建混合监督与遏制基础设施
AI 实时监控 + 自动遏制协议 + 人类专家审查的三层系统。基于不可变账本和标准化审计追踪，为自动遏制和人工裁决提供可验证记录
4
在监管沙盒中开展试点项目
纯理论方法不够充分。创建受监督的实验室，让私营企业、学术研究人员和监管机构合作部署、观察小规模代理经济，积累实证数据
5
投资劳动力互补与现代化社会安全网
双支柱策略：系统性重塑教育培养人类与 AI 协作能力（批判性思维、创造力）+ 强化社会保障（适应性失业保险、可携带福利、负所得税）
Conclusion

嵌入社会目标于代理经济基础设施

通过将社会目标嵌入代理间交易的基础设施，我们可以培育一个以涌现协作为特色而非缺陷的生态系统
