01 · 协议介绍
What is A2A?

什么是 A2A Protocol

Agent2Agent（A2A）Protocol 是一个开放标准，旨在实现不同 AI Agent 之间的无缝通信与协作。 它为构建在不同框架上、由不同供应商开发的 Agent 提供了通用语言。

🤖

核心问题

AI Agent 通常由不同开发者、不同框架、部署在不同组织，彼此难以直接协作，形成信息孤岛。

🔓

A2A 的解法

提供标准化通信协议，让来自不同来源的 Agent 联合起来，共同完成复杂任务，打破信息孤岛。

📋

技术基础

复用 HTTP、JSON-RPC 2.0、Server-Sent Events 等成熟标准，降低学习成本和实现难度。

🏛️

治理

最初由 Google 开发，现已捐赠给 Linux 基金会进行维护和管理，保持开放中立。

01 · 设计原则
Design Principles

五大核心设计原则

⚡

简单性 Simplicity

复用 HTTP、JSON-RPC 2.0、Server-Sent Events 等经过广泛验证的标准，降低学习成本和实现难度。

🏢

企业级就绪 Enterprise-Ready

充分考虑身份验证、授权、安全性、隐私保护、追踪和监控等企业级需求，与成熟实践保持一致。

⏳

异步优先 Async-First

专为长时间运行的任务和 Human-in-the-Loop 交互而设计，能够优雅处理长时间运行的操作。

🎨

模态无关 Modality-Agnostic

支持多种内容类型：文本、音频/视频（文件引用）、结构化数据/表单，乃至嵌入式 UI 组件。

🔒

不透明执行 Opaque Execution

Agent 基于声明的能力和交换的信息协作，无需共享内部思考过程、计划或工具实现细节。

02 · 基本概念
Core Participants

三种核心参与者

👤

User（用户）

最终终端用户，通过与 A2A Client 交互发起请求。不直接参与 A2A 协议通信。

📡

A2A Client（客户端）

代表用户或其他系统发起请求的应用程序或 Agent。充当协调器，调度和整合多个远程 Agent 的能力。

🖥️

A2A Server（远程 Agent）

暴露 A2A 兼容端点的 Agent 或 Agentic 系统。发布 Agent Card，处理任务，提供响应。

User → 发起请求 → A2A Client → A2A 协议 → A2A Server （可继续调用其他 Server）
02 · 通信元素
Communication Elements

五大通信元素

🪪 Agent Card

Agent 的数字名片，JSON 格式元数据。包含身份、端点、能力列表、认证要求。发布于 /.well-known/agent-card

💬 Message

一次通信回合的基本单位。含唯一 messageId、角色标识（user/agent），承载一或多个 Part。

📋 Task

核心工作单元，具有唯一 ID 和清晰生命周期。用于长时间运行操作，通过 contextId 分组形成会话。

🧩 Part

最基本的内容容器，使用 oneof 结构。类型：TextPart / FilePart / DataPart

📦 Artifact

任务处理中生成的有形交付物（文档、图像、代码等）。含唯一 artifactId，支持增量流式传输。

02 · 交互机制
Interaction Mechanisms

三种交互模式

⚡ 同步消息

适用于即时、自我包含的交互，Agent 可立即返回结果，无需维护复杂状态。

→sendMessageClient 发送请求
←Message 对象直接返回

🌊 流式传输

通过 sendMessageStream 实现，允许实时接收任务更新。

→sendMessageStream
←Task 对象（开始）
←StatusUpdateEvent×N
←ArtifactUpdateEvent×N
←final=true（结束）

📨 推送通知

长时间运行或断开连接场景，通过 webhook 实现服务器主动推送通知。

→配置 webhook URL
←Task 状态变化时自动推送
←StreamResponse 格式

03 · 通信流程
Communication Flow

五步通信流程

1

Agent 发现

客户端通过 HTTP GET 请求获取目标 Agent 的 Agent Card，通常发布在 /.well-known/agent-card 端点。

2

身份验证

解析 securitySchemes 字段，根据 authorizationUrl 和 tokenUrl 向授权服务器请求 JWT 令牌。

3

发送消息

根据 Agent Card 中的 API 端点，发送 POST 请求（携带 JWT 令牌），请求体包含 Message 对象及任务描述。

4

处理与响应

Agent 可能创建 Task（异步模式）或直接返回 Message（同步模式）。异步任务在后台继续处理。

5

接收更新

三种方式获取异步任务更新：轮询（Get Task）· 流式订阅（持久连接）· 推送通知（webhook）

04 · 生命周期
Lifecycle & Context

多轮交互与上下文管理

🔑 contextId 的作用

逻辑分组多个 Task 和 Message，提供连续性。服务器负责生成；客户端提供则尽量复用。用于管理内部 LLM 上下文，确保多轮对话连贯性。

🔄 任务不可变性原则

任务达到终态后不可重启。后续细化须在同一 contextId 内发起新任务，可通过 referenceTaskIds 引用原任务。

💬 INPUT_REQUIRED 交互

Agent 需要额外信息时将状态设为 INPUT_REQUIRED，客户端检测后发送包含所需信息的跟进消息，任务继续处理。

⚡ 并行跟进

同一 contextId 内支持并行任务。客户端可在前置任务完成后立即创建依赖任务，实现高效并行工作流。

📦
产物版本管理（客户端驱动）
Agent 不维护产物变更历史，客户端负责追踪。服务 Agent 在生成细化版本时应使用一致的 artifact-name，方便客户端追踪最新版本。
